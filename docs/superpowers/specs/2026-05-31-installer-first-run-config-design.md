# Installer first-run configuration — API keys + persona

- **Date:** 2026-05-31
- **Status:** Design approved; pending spec review
- **Branch:** feat/jarvis-talking-face
- **Soul UX:** Option A (copy → edit) — switchable to B (guided, no editor)

## Problem

The all-in-one installer (`install.sh`) installs all four channels but does **not**
help the user provide API keys (it only writes an *empty* `.env` template via
`setup_env_template`) or personalize the assistant (the soul is never touched).
A fresh user therefore lands on a non-working voice agent and must hand-edit files
they don't yet know exist (`.env`, `src/voice-agent/.env`, `~/.jarvis/SOUL.md`).

## Goal

Extend the **single** all-in-one `install.sh` (must keep working via both
`curl … | bash` and `./install.sh`) so that, after the deps/builds finish, it
interactively:

1. Collects API keys per provider and writes them to the correct files.
2. Personalizes the assistant's soul, writing `~/.jarvis/SOUL.md`.

Both steps degrade gracefully to the current non-interactive behavior (template +
printed instructions) when no terminal is reachable. Nothing ever hangs.

## Non-goals

- No separate `jarvis setup` command — it must be part of the one installer.
- No changes to services/systemd, the git-tracked `prompts/soul.md`, or LiveKit
  key generation (already automatic via `setup_livekit_keys`).
- No network validation of provider keys (just non-empty capture).

## Architecture — where it hooks in

A new `configure()` stage invoked from `main()` after the build/install stages and
after `generate_bridge_token` / `ensure_livekit_binary` / `setup_livekit_keys`.
It **subsumes** today's `setup_env_template` (which becomes the non-interactive
fallback writer). Sub-functions, each with one job:

- `_interactive()` — gate helper.
- `configure_api_keys()` — provider key capture + write.
- `configure_soul()` — persona copy/personalize + write `~/.jarvis/SOUL.md`.
- `_env_upsert <file> <VAR> <value>` — idempotent `.env` line writer.

## Interactivity model

- `_interactive()` is true iff a terminal is reachable — `[ -t 0 ]` **or**
  `/dev/tty` is read/writable — **and** `JARVIS_NONINTERACTIVE != 1` **and** not
  `JARVIS_DRY_RUN`.
- All prompts read/write through `/dev/tty`, so `curl … | bash` (whose stdin is the
  script itself) can still prompt the real terminal.
- Non-interactive (CI / headless / piped-without-tty / `JARVIS_NONINTERACTIVE=1`):
  skip prompts, write the commented `.env` template (today's behavior), print exact
  next-step instructions naming the files to edit.
- New skip flag `JARVIS_SKIP_SETUP=1` bypasses the whole `configure` stage
  (deps/builds still run), mirroring the existing `JARVIS_SKIP_*` family.

## Component 1 — `configure_api_keys()`

Provider table → `{ env var, target file }`:

| Provider | Env var | Target file |
|---|---|---|
| Anthropic | `ANTHROPIC_API_KEY` | repo-root `.env` |
| Groq | `GROQ_API_KEY` | repo-root `.env` |
| OpenAI | `OPENAI_API_KEY` | repo-root `.env` |
| DeepSeek | `DEEPSEEK_API_KEY` | repo-root `.env` |
| Google | `GOOGLE_API_KEY` | repo-root `.env` |
| Kimi | `KIMI_API_KEY` | repo-root `.env` |
| Deepgram (streaming STT) | `DEEPGRAM_API_KEY` | `src/voice-agent/.env` |

LiveKit keys are **not** prompted (auto-generated by `setup_livekit_keys`).

Flow:
1. Prompt the live defaults first — **Anthropic, Groq, Deepgram**.
2. Then a single gate: "Configure more providers? [y/N]" → OpenAI, DeepSeek,
   Google, Kimi. Keeps it from being a 7-prompt slog.
3. Per key: hidden input (`read -s … < /dev/tty`); blank = skip (leave
   unset / keep existing). If the var already has a non-empty value in the target
   file → "`<VAR>` is already set — replace? [y/N]".
4. `_env_upsert` replaces the existing `^VAR=` line if present, else appends,
   preserving all other lines; `chmod 600` the file afterward.
5. After the pass: if **no** LLM key is set anywhere, print a clear warning ("the
   voice agent needs at least one LLM key — set one in `<file>` before starting")
   but do **not** fail the install.

Idempotent: re-running offers replace; never silently clobbers.

## Component 2 — `configure_soul()` (Option A: copy → edit)

1. If `~/.jarvis/SOUL.md` exists → "It already exists — overwrite? [y/N]" (default
   keep). Else → "Personalize the assistant now? [Y/n]" (no = skip; JARVIS uses the
   git `soul.md`).
2. On yes: `mkdir -p ~/.jarvis`; copy `src/voice-agent/prompts/soul.md` →
   `~/.jarvis/SOUL.md`; `chmod 600`.
3. Optional quick-set: prompt for the assistant's spoken **name** (default keep
   "JARVIS"); if changed, apply a conservative, documented substitution of the
   canonical identity references only (not arbitrary text).
4. Open `$EDITOR` (fallback `nano` → `vi`) on `~/.jarvis/SOUL.md` for hand tweaks.
   Non-interactive: skip the editor; leave the copied (optionally name-applied)
   file and print "edit `~/.jarvis/SOUL.md` to customize".

**Load constraint (resolved — `prompt_builder.py` untouched):**
`~/.jarvis/SOUL.md` is loaded by `pipeline/prompt_builder.py::load_soul` as
**untrusted** input — injection-scanned (`_scan_soul_override`) then truncated to
`MAX_SOUL_CHARS = 40000`. The canonical `prompts/soul.md` is **30,331 chars**, so a
full copy fits with ~9.6k of headroom for the user's edits before truncation — **no
`prompt_builder.py` change needed.** Caveat: the copied file still passes through the
injection scan; the hand-curated soul is expected to pass, and a scan block *fails
safe* (load_soul falls back to the git `soul.md`, so JARVIS still works — it just
ignores the override). The wizard warns if the final file trips the scan or nears
the 40k cap.

## Error handling

- Every prompt is skippable; nothing aborts the install.
- File writes are best-effort; failures `warn` (e.g. can't write `~/.jarvis`) and
  continue.
- `set -euo pipefail`-safe: all new env reads use `${VAR:-default}` (no unguarded
  vars — same class as the `install_bubblewrap` bug just fixed in `c0dfe682`).

## Testing

- `shellcheck -S error install.sh` + `bash -n install.sh`.
- **Non-interactive path:** run `configure` with `JARVIS_NONINTERACTIVE=1` and a
  throwaway `HOME` / repo copy; assert the template is written and the run does not
  hang.
- **Interactive path:** drive prompts by feeding scripted answers to `/dev/tty`
  (pty via `script`/`expect`, or a `/dev/tty` shim) under a temp `HOME`; assert keys
  land in the right files at `chmod 600` and `~/.jarvis/SOUL.md` is created.
- **Soul load:** call `load_soul()` with the temp `HOME` and assert it returns the
  generated `SOUL.md` content (not truncated away).
- All tests use a throwaway `HOME` and temp clone — never touch the live
  `~/.jarvis`, `.env`, or services.

## Scope

- **IN:** `install.sh` only — `configure`, `configure_api_keys`, `configure_soul`,
  `_interactive`, `_env_upsert`.
- **OUT:** `prompt_builder.py` (base soul fits the 40k cap — no change),
  services/systemd, git `prompts/soul.md`, LiveKit key generation, `src/cli/`,
  web/desktop code.
